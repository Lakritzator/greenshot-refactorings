# Translation System

This page explains how to use the `Greenshot.Translations` library to localise the
application and its plugins.

---

## Overview

Greenshot.Translations is a lightweight, reflection-free localisation library.
Translations are stored in simple INI-style text files.
The active language and all registered plugin translations are managed through a
single `ITranslationManager` instance that the host application creates at startup
and passes to every plugin during initialisation.

---

## File format

Translation files are plain UTF-8 text files with the extension `.ini`.

```ini
# Lines starting with # or ; are comments — they are ignored.
# Blank lines are also ignored.

cancel=Cancel
ok=OK
about_bugs=Please report bugs to

# Use \n for embedded newlines, \t for a tab, \\ for a literal backslash.
about_license=Copyright © 2004-2026 Thomas Braun, Jens Klingen, Robin Krom\nLine 2 of the license.
```

Rules:
- Each data line has the form `key=value`.
- The key is trimmed; everything after the first `=` is the raw value.
- `\n` → newline, `\t` → tab, `\\` → single backslash.
- The language display name (e.g. "English") is resolved at runtime from
  .NET's `CultureInfo.NativeName` — it does not need to appear in the file.
- There is no version field — translation files are versioned with the source code.

---

## File naming convention

| Scope | Filename pattern | Example |
|-------|-----------------|---------|
| Main application | `{ietf}.ini` | `en-US.ini`, `de-DE.ini` |
| Plugin | `{prefix}.{ietf}.ini` | `box.en-US.ini`, `imgur.de-DE.ini` |

The IETF language tag and optional plugin prefix are derived entirely from the
filename.  Both pieces of information are available without reading the file body,
so `GetAvailableLanguages()` is very fast — it only calls `Directory.GetFiles`.

---

## Directory layout

```
Languages/
├── en-US.ini          ← main application, English
├── de-DE.ini          ← main application, German
├── fr-FR.ini
├── box.en-US.ini      ← Box plugin, English
├── box.de-DE.ini
└── imgur.en-US.ini
```

Each plugin places its files in its own `Languages/` sub-folder inside the plugin
installation directory.  The host application and all plugins share a single
`ITranslationManager` instance, but each registers its own directory.

---

## Setup — host application

Create the `IniTranslationProvider` once at application startup, register the
application's language directory, and set the initial language.

```csharp
using Greenshot.Translations.Core;
using Greenshot.Translations.Wpf;   // only needed for WPF

// 1. Create the shared provider
var manager = new IniTranslationProvider();

// 2. Register the host application's language directory
manager.AddPath(Path.Combine(AppContext.BaseDirectory, "Languages"));

// 3. Set the fallback language (used when a key is not found)
manager.FallbackLanguage = "en-US";

// 4. Set the initial language
manager.CurrentLanguage = "en-US";

// 5. (WPF only) Make the manager available to markup extensions
TranslationManagerLocator.Current = manager;

// 6. Pass manager to each plugin during plugin initialisation
foreach (var plugin in LoadedPlugins)
{
    plugin.Initialize(manager);
}
```

## Setup — plugin

A plugin receives the shared `ITranslationManager` and calls `AddPath` to register
its own language files.  No other registration is required.

```csharp
public class BoxPlugin
{
    private IBoxTranslations _translations;

    public void Initialize(ITranslationManager translationManager)
    {
        // Register the plugin's language directory
        var languagesDir = Path.Combine(_pluginFolder, "Languages");
        translationManager.AddPath(languagesDir);

        // Create the typed translation object for this plugin
        _translations = new BoxTranslations(translationManager);
    }

    private void OnUploadClick(object sender, EventArgs e)
    {
        MessageBox.Show(_translations.UploadMenuItem);
    }
}
```

`BoxTranslations` is generated by the source generator from `IBoxTranslations` —
you never write it by hand.

## Defining a translation interface

Annotate an interface with `[TranslationGroup]` and list one property per key.
The source generator creates the implementation class automatically.

### Main application

```csharp
using Greenshot.Translations.Attributes;

namespace Greenshot.Configuration;

[TranslationGroup]                          // no prefix = main application
public interface IGreenshotTranslations
{
    string Cancel { get; }                  // → key "cancel"
    string Ok { get; }                      // → key "ok"
    string AboutBugs { get; }              // → key "about_bugs"
    string SettingsTooltipLanguage { get; } // → key "settings_tooltip_language"
}
```

### Plugin

```csharp
[TranslationGroup("box")]                   // prefix "box"
public interface IBoxTranslations
{
    string Configure { get; }              // → key "box.configure"
    string UploadMenuItem { get; }         // → key "box.upload_menu_item"
    string UploadSuccess { get; }          // → key "box.upload_success"
}
```

Key mapping rule: `PascalCase` → `snake_case`.
`SettingsTooltipLanguage` → `settings_tooltip_language`.

## Using translations

### Typed access (recommended)

```csharp
// Constructed once, passed by DI or stored as a field
var translations = new GreenshotTranslations(manager);

cancelButton.Text = translations.Cancel;
```

The generated class implements `INotifyPropertyChanged`, so WPF/Avalonia bindings
refresh automatically when `CurrentLanguage` changes.

### String-key access

```csharp
string text = manager.GetString("cancel");

if (manager.TryGetString("about_bugs", out var value))
{
    label.Text = value;
}
```

Missing keys return `###key###` rather than throwing.

## WPF usage

```xml
<Window xmlns:t="clr-namespace:Greenshot.Translations.Wpf;assembly=Greenshot.Translations">
    <StackPanel>
        <Button Content="{t:Translate cancel}" />
        <Label Content="{t:Translate about_bugs}" />
    </StackPanel>
</Window>
```

The `{t:Translate}` extension creates a `Binding` to the `TranslationIndexer`
singleton, so controls update automatically on language change.

Alternatively, set a generated translations object as `DataContext`:

```xml
<!-- Translations class set as DataContext in code-behind -->
<Button Content="{Binding Cancel}" />
```

## Windows Forms usage

Mark each control with a `TranslationKey` using `ITranslatable`, then call
`FormsTranslationHelper.ApplyTranslations` once after `InitializeComponent`.

```csharp
public class SettingsForm : Form
{
    private readonly TranslatableButton _btnCancel = new()
    {
        TranslationKey = "cancel"
    };

    public SettingsForm(ITranslationProvider provider)
    {
        InitializeComponent();
        Controls.Add(_btnCancel);

        // Apply now and re-apply automatically on language change
        FormsTranslationHelper.ApplyTranslations(this, provider);
    }
}
```

`FormsTranslationHelper` uses thread-safe `BeginInvoke` when re-applying
after a language change, so no manual marshalling is required.

## Listing available languages

```csharp
foreach (var tf in manager.GetAvailableLanguages())
{
    Console.WriteLine($"{tf.DisplayName} ({tf.Ietf})");
    // "English (United States) (en-US)"
    // "Deutsch (Deutschland) (de-DE)"
}
```

`TranslationFile.DisplayName` is resolved from `CultureInfo.NativeName`.
`TranslationFile.Prefix` is `null` for main-application files and a string
(e.g. `"box"`) for plugin files.

---

## Switching language at runtime

```csharp
manager.CurrentLanguage = "de-DE";
// Fires LanguageChanged, clears the cache, reloads all registered directories
```

All WPF bindings created by `{t:Translate}` refresh automatically.
Windows Forms controls registered with `FormsTranslationHelper` also refresh
automatically via the `LanguageChanged` subscription.

---

## Dialect fallback chain

When a requested language is not fully available, the library falls back
progressively from most-specific to least-specific:

```
zh-TW  →  zh  →  en-US  (FallbackLanguage)
de-x-franconia  →  de  →  en-US
nn-NO  →  nn  →  en-US
```

The fallback language (`en-US` by default) is always loaded first so that any
key missing from the selected language is still displayed in English rather than
as a `###key###` sentinel.

## Creating a new translation

1. Copy `en-US.ini` (or the relevant plugin file) to a new file named after the
   target language, e.g. `fr-FR.ini`.
2. Translate only the values — keep all keys unchanged.
3. Preserve `\n` escape sequences wherever they appear.

### LLM prompt

The following prompt can be used with any LLM to translate a file:

```
You are a professional software translator.

Translate the following INI translation file from English to {TARGET_LANGUAGE}.
Keep all keys (the left-hand side of each = sign) unchanged.
Translate only the values (the right-hand side).
Preserve \n newlines and \t tabs exactly as they appear.
Keep all comment lines (starting with #) unchanged.
Return the complete translated file.

--- SOURCE (en-US.ini) ---
{PASTE FILE CONTENT}
--- END SOURCE ---

Output the complete translated INI for {TARGET_LANGUAGE} ({IETF_CODE}).
```
